## 리뷰
왜 branch를 나누라고 했는지 확실히 알았다. 지금까진 W1M1 브랜치에서 곧바로 `git checkout -b W1M2`를 실행해서, W1M1에서 작업한 내용이 그대로 W1M2에 전달되었는데, 이렇게 하지 않고 W1M1 작업이 끝났으면 main 브랜치로 돌아가서  `git checkout -b W1M2`를 실행했어야 했다. W2부턴 이런 식으로 branch를 나눠서 미션별로 작업을 정리하고자 한다.  
W1M3에서, usd_Million 컬럼을 놔둔 채로 이 값을 1000으로 나눈 usd_Billion을 만들었는데, 이렇게 하면 usd_Million을 1000으로 나누어 저장하는 도중 작업이 중단된다면 롤백은 수월하겠지만 비슷한 성격의 중복된 내용이 테이블의 공간을 차지한다는 단점이 있다. 따라서 usd_Billion 컬럼이 있는 테이블의 경우, usd_Million 컬럼을 삭제했다.  
오늘 오전 수업을 듣고 내가 미처 생각하지 못한 부분을 고민해보는 시간을 가지며 W1M3 코드를 수정했다. 첫째, url 경로, logfile 경로 등의 data를 code와 분리했다. 둘째, usd_Million 컬럼이 들어있는 원본 데이터는 Countries_by_GDP.json에, usd_Billion 컬럼과 Region 컬럼이 들어있는 정제된 데이터는 Countries_by_GDP_Region.json에 저장했다.  

## 회고
Keep(계속 유지했으면 좋을 과정): 미션 요구사항 부분에서 다양한 해석이 나오는 부분은 팀원과 토의하며 어떤 해석이 가장 합리적일지 도출하기  
Problem(문제가 발생한 과정): 문제가 요구한 상황에 맞게만 코드가 돌아갈 수 있도록 미션을 진행함 → 문제가 요구한 상황 외의 맥락을 고려하며 코드를 수정함(데이터를 가져온 웹의 HTML 구조가 변경되었을 경우 어떻게 해야 코드를 빠르게 유지보수할 수 있을지, 테이블 구조를 어떻게 해야 롤백이 가능하고 연산이 빨라질지 등)  
Try(다음번에 시도해보면 좋을 과정): 팀 활동 시, 코드 레벨 단위에서 팀원과 논의해보는 시간 가지기. 수도코드를 우선적으로 만들어놓고 팀원과 로직을 논의하고, 나중에 완성된 코드와 수도코드를 비교해보기
